### 9. Подготовка к интерпретации построенных моделей

**9.1 Используемые паттерны проектирования и разработки [✋](https://github.com/stankin/design-part-2/wiki/sem2):**

* **Процессная модель для сравнения:**

Задача: создание системы для наиболее эффективного контроля состояний производственных процессов.

**Решение задачи при помощи методологии PDCA:**

1. **Plan (Планирование):**

 + Выявленные проблемы :

   + планово-профилактическоий ремонт производственного оборудования;
   + отсутсвие возможности предсказывать поломки производственного оборудования;
   + анализ истории поломок в контексте производственного процесса занимает много времени;

 + Цели: 

   + сокращение временных затрат на анализ поломок;
   + обслуживание по фактическому состоянию;
   + предсказание поломок;
   + формирование отчетов и ведение мониторинга. 

 + Требования: 
   + повысить эффективность без увеличения трудозатрат;
   + конечная система должна быть проста в использовании всех ее пользователей.

 + Ожидаемый результат: Аналитическое программное средство для эффективного обслуживания по фактическому состоянию.

 + Ресурсы, необходимые для достижения ожидаемого результата: 

   + IIoT-инфраструктура (Контроллеры состояний производственного оборудования, информационные базы данных и знаний);
   + Веб-сервер;
   + Клиент; 

 + Процессы (запланированные действия):

   + Проектирование;
   + Верстка;
   + Моделирование;
   + Fullstack - разработка системы;
   + Ручное тестирование. 

2. **Do (Выполнение)** : Выполнение поставленных задач сотрудниками, обладающими правами на использование системы. 

3. **Check (Проверка)** : Оценка эффективности на основе результатов внедрения и использования системы. 

4. **Act (Улучшение)** : При успешной работе системы (при сокращении временных и финансовых затрат на обслуживание оборудования), он внедряется в информационную систему предприятия, с возможностью доработки или расширения функционала в дальнейшем. 

* **Используемые в разработке паттерны и фреймворки:**
  + Фреймворк: Bootstrap 5
  + Фреймворк: Django
  + Паттерн: Model-View-Controller

**9.2 Используемые паттерны выявления проблем [✋](https://github.com/stankin/design-part-2/wiki/sem3):**
* Муда: ведение учета состояний вручную. 
* Мура: составление план-графика работ раз в месяц. 
* Мури: один технолог на 3 производственных процесса. 

**9.3 Возможные антипаттерны [✋](https://github.com/stankin/design-part-2/wiki/sem4):**

| Категория  | Антипаттерн (риск) | Действие по избежанию |
|---|---|---|
| Разработка | Таинственный код (Cryptic code) - использование аббревиатур вместо понятных имен | Задания имён переменным и функциям, в соотвествии с их назначением |
| Архитектура | Большой комок грязи (Big ball of mud) - программа с нераспознаваемой структурой | На этапе проектирования чётко определить структуру программы по паттерну MVC |
| Организация | Раздутый улучшизм (Creeping featurism) - добавление новых улучшений в ущерб суммарному качеству программы | Добавление только тех функций, которые действительно необходимы |
| Среда | Увлечение модными словами (Buzzword Mania) - руководство жонглирует словами, которые мало кто из подопечных понимает | Использование простых формулировок при общении с сотрудниками |

Одним из важных числовых показателей являются время и трудозатраты на принятие решения. 

После внедрения аналитического ПС минимизации простоев производственных процессов, на принятие решения о необходимости проведения обслуживания по фактическому состоянию потребуется 2 минуты + 1 минута на передачу отчета на производство. Итого - 3 минуты. 

**10.3 Расчет потенциального эффекта от автоматизации:**

Основным недостатком стратегии планово-профилактического ремонта является проведение излишних мероприятий по обслуживанию оборудования – в ходе профилактики велика вероятность проведения осмотров фактически исправного оборудования. Эта проблема касается не только профилактических мероприятий, но и ремонтов, т. к.  все ремонты проводятся в соответствии с регламентированным расписанием. Если на один производственный процесс из 5 единиц производственного оборудования назначить проведение полного осмотра по 2 часа на каждую единицу раз в месяц, то один технолог потратит 10 часов на выполнение этой работы + время на ремонт - 4 часа. Весь производственный процесс будет неработоспособен на 14 часов в месяц. 

После внедрения аналитической системы и перехода на обслуживание по фактическому состоянию полного осомтра всех единиц оборудования производственного процесса не потребуется. При еженедельном анализе отчетности по 1 часу проведение осмотра и ремонта займет до 6 часов при выявлении проблемы в отчете. Отсутсвие плана на осмотр не потребует остановки процессов ежемесячно / еженедельно. На все задачи потребуется до 4 часов в неделю. Таким образом экономия времени составит 14ч - 4ч = 10ч в месяц. При продолжительности смены 8 часов ежемесячные затраты составят 4/8*1 = 0.5 чел/месяц
